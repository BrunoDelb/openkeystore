Air-tight Provisioning and Keystore API for the SKS


SKS = Secure Key Store; a smart card that offers end-to-end secured key-provisioning and key-management.


General:

There is a not shown 2-byte status code returned by all methods. In case the status is <> 0 there is an error and the succeeding values MUST NOT be read as they are not available. Instead there is a second return value containing an UTF-8 encoded description in English to be used for logging and debugging purposes.


Provisioning Session Management

attested_provisioning_report = closeProvisioningSession (secure_provisioning_handle, assumed_result, mac)


This method SHOULD commit all accumulated provisioned data if the assumed_result matches the actual state. The attested_provisioning_report is simply the MAC-signed value of the word “Success” if the operation is successful, otherwise the method returns an error code + string. In case there is a problem all associated provisioned data SHOULD be removed (roll-backed).

Note: if the provisioning handle is valid, all session data will be deleted.

Note: this method is only allowed on a session that is still open.

Note: this method enumerates both closed and open provisioning sessions.


setTime (milliseconds_since_epoch)


This method SHOULD be issued before any other method for a newly created SKS instance.



Provisioning Session Object Creation


PUK_index = createPUKPolicy (secure_provisioning_handle, puk_policy_data...TBD, encrypted_PUK, hash_of_sorted_enclosing_key_ids)


A PUK with RetryLimit = 0 means it never expires but always introduces a 1-10 second delay before each usage (correct or incorrect)


PIN_index = createPINPolicy (secure_provisioning_handle, PUK_index, pin_policy_data...TBD, hash_of_sorted_enclosing_key_ids)


PUK_index = 0 means device PUK. Device PUKs may not be available on all keystores and in that case PUK_index = 0 means no PUK.

Setting Data to Current Provisioning Session Created Objects

The mac is a TBD combination of the operation, the session key, the target key data, and the argument data.


setCertificatePath (secure_provisioning_handle, local_key_index, mac, certificate_path {,friendly_name})


Note: the SKS MUST NOT verify that the certificate_path and the public key match for keys having the “import_private_key” flag set because that would disable the restorePrivateKey method. For other keys, the SKS MAY perform such a test (although it is redundant since the mac is assumed to cater for the binding between certificate data and the generated public key). I.e. a conforming SKS MAY always treat certificate_path data as an array of “blobs”.


The certificate_path argument must be represented in such a way that the SKS can identify the individual certificates as well as being sorted with the EE-certificate at the beginning of the structure.


A conforming SKS MUST verify that exactly one such operation is performed towards each created key in a session.


A conforming SKS MUST verify that this is the first operation that modifies the created key in a session because the mac of possible succeeding operations is built on the EE-certificate, not the public key.



addExtension (secure_provisioning_handle, local_key_index, mac, extension_data, URI)


Note: in order to verify integrity of the provisioning session Extension objects MUST be locally provisioned in the same order as the issuer created them because the mac also includes the mac of the preceding item with an array of zero bytes as an initial (dummy) value.


addLogotype (secure_provisioning_handle, local_key_index, mac, image_data, MIME_type, URI)


Note: in order to verify integrity of the provisioning session Logotype objects MUST be locally provisioned in the same order as the issuer created them because the mac also includes the mac of the preceding item with an array of zero bytes as an initial (dummy) value.


setPiggybackedSymmetricKey (secure_provisioning_handle, encrypted_symmetric_key, mac, local_key_index, endorsed_algorithms_in_local_notation[])


Note: a conforming SKS SHOULD verify that only one such operation is optionally performed towards a specific local_key_index in a session.


restorePrivateKey (secure_provisioning_handle, encrypted_private_key, mac, local_key_index)

Restrictions on local_key_index_master:

MUST be PIN-protected.

Restrictions on local_key_index_copy:

MUST be fully provisioned.

MUST NOT be PIN-protected.
